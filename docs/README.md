## 详细设计

如果你要真正实现一个框架，不要只读 00-30 的总览章节，先进入 [详细设计索引](./详细设计/00-索引.md)。详细设计目录按专题写清楚了状态机、阈值、数据结构、伪代码、失败分支、提示词模板、测试用例和验收标准。

详细设计中的源码提示词必须标注来源：来自 Claude Code 源码的提示词会同时给出英文原文、中文版本、源码路径和调用路径；不是源码内置的模板会明确标为本框架建议模板。

## 参数解释规范

任何参数不得只写名字和默认值。读者第一次看到一个参数时，必须同时知道它是什么、单位是什么、为什么是这个值、什么时候触发、调大或调小会造成什么后果。

所有重要参数表统一使用这些列：

| 参数名 | 中文含义 | 单位 | 默认值 | 为什么是这个值 | 触发行为 | 调大后果 | 调小后果 |
|---|---|---|---:|---|---|---|---|
| `exampleLimit` | 示例限制值 | 次 | `3` | 3 次能覆盖偶发失败，同时避免无限循环。 | 连续失败达到 3 次时熔断。 | 更容易恢复偶发问题，但可能浪费成本。 | 更快失败，但可能把偶发问题暴露给用户。 |

所有公式必须解释每个变量。只给公式不合格。例如 `effectiveWindow = contextWindow - min(modelMaxOutputTokens, 20_000)` 必须同时说明：

- `contextWindow`：目标模型总上下文窗口。
- `modelMaxOutputTokens`：本次调用给模型预留的最大输出。
- `20_000`：输出/压缩摘要预留上限，防止 prompt 占满窗口。
- `effectiveWindow`：真正允许 prompt 使用的最大空间。

所有阈值必须说明达到阈值后的动作。比如 warning 只提示，auto compact 会自动压缩，block 会禁止直接调用模型。
